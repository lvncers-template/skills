# Grill with Docs

最終更新: [e74f006](https://github.com/mattpocock/skills/commit/e74f0061bb67222181640effa98c675bdb2fdaa7)

既存のドメインモデルに対してプランを徹底的に検証し、用語を精査し、決定事項が固まるたびにドキュメント（CONTEXT.md、ADR）をその場で更新する。
プロジェクトの言語と文書化された決定事項に照らしてプランをストレステストしたいときに使う。

このプランのあらゆる側面について、共通理解に達するまで容赦なくインタビューする。
設計ツリーの各ブランチを順番にたどり、決定事項の依存関係を一つずつ解決していく。各質問には推奨する回答も提示する。
質問は一度に一つずつ行い、次の質問に進む前に各質問へのフィードバックを待つ。
コードベースを探索することで答えられる質問は、探索して答える。

## ドメインの把握

コードベースを探索する際は、既存のドキュメントも確認する。

### ファイル構成

ほとんどのリポジトリはコンテキストが一つ：

```sh
/
├── CONTEXT.md
├── docs/
│ └── adr/
│ ├── 0001-event-sourced-orders.md
│ └── 0002-postgres-for-write-model.md
└── src/
```

ルートに `CONTEXT-MAP.md` が存在する場合、リポジトリは複数のコンテキストを持つ。

## セッション中の進め方

### 用語集との照合

ユーザーが `CONTEXT.md` の既存の言語と矛盾する用語を使った場合、すぐに指摘する。

### 曖昧な言葉を精査する

ユーザーが曖昧な言葉や多義的な言葉を使った場合、正確な標準用語を提案する。

### 具体的なシナリオで議論する

ドメインの関係性を議論するときは、具体的なシナリオでストレステストを行う。

### コードとの整合性確認

ユーザーが何かの動作を述べた場合、コードがそれと一致しているか確認する。

### CONTEXT.md のリアルタイム更新

用語が確定したら、その場で `CONTEXT.md` を更新する。まとめて後回しにしない。
`CONTEXT.md` には実装の詳細を一切含めない。あくまで用語集のみ。

### ADR は慎重に提案する

以下の三つすべてが揃ったときだけADRの作成を提案する：

1. **覆しにくい** — 後で考えを変えるコストが大きい
2. **文脈なしでは驚く** — 将来の読者が「なぜこうしたのか？」と思うような内容
3. **本物のトレードオフの結果** — 真剣に検討した選択肢があり、特定の理由で一つを選んだ
